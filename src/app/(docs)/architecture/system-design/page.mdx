## Core Components

### Memory Manager

Central orchestrator handling all CRUD operations. Routes operations to appropriate memory tiers and manages cross-tier consistency.

### Retrieval Engine

Hybrid search coordinator. Dispatches queries to vector and text indices, then fuses results using configurable ranking algorithms.

### Sleep Consolidator

Background worker running consolidation cycles. Compresses, promotes, extracts, and evicts entries according to retention policies.

### Index Manager

Maintains all database indices: vector embeddings (via `sqlite-vec`), FTS5 full-text, and temporal graph relationships.

## Technology Stack

| Layer | Technology | Purpose |
|-------|-----------|---------|
| Database | SQLite 3.45+ | Primary storage |
| Vectors | sqlite-vec | Vector similarity search |
| Text | FTS5 | Full-text search |
| Embeddings | OpenAI / SentenceTransformers | Text-to-vector |
| ORM | SQLAlchemy | Schema management |
| API | FastAPI | REST endpoints |

## Scaling Considerations

Mnemosyne is designed for single-node deployments:

- **Working Memory**: In-memory LRU with SQLite backing
- **Episodic Memory**: SQLite with vector extension
- **Semantic Memory**: Native graph in SQLite

For multi-agent deployments, each agent maintains its own database instance. Shared memory is achieved via explicit synchronization, not concurrent database access.

<Callout type="warning" title="Not a Distributed System">
Mnemosyne does not support distributed or clustered deployments. For high-availability requirements, use SQLite replication (Litestream) or export/import between instances.
</Callout>
