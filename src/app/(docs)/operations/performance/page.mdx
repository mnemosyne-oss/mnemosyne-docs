# Performance

Optimize Mnemosyne for your workload.

## Benchmarks

All benchmarks run on a single core, SQLite with `sqlite-vec`:

| Metric | 1K Entries | 10K Entries | 100K Entries |
|--------|-----------|-------------|--------------|
| Write | 12ms | 15ms | 18ms |
| Vector search | 15ms | 35ms | 85ms |
| Hybrid search | 45ms | 65ms | 150ms |
| FTS5 search | 5ms | 8ms | 20ms |
| Consolidation | 2s | 5s | 25s |

## Optimization Strategies

### 1. Batch Writes

<CodeBlock code={`# Bad: Individual writes
for item in items:
    mem.remember(item)

# Good: Batch transaction
with mem.transaction():
    for item in items:
        mem.remember(item)
`} language="python" />

### 2. Enable WAL Mode

<CodeBlock code={`# Enable WAL for better concurrent read/write performance
sqlite3 mnemosyne.db "PRAGMA journal_mode=WAL;"

# Or in Python
mem = Memory(wal_mode=True)
`} language="python" />

### 3. Embedding Caching

Cache embeddings to avoid redundant API calls:

<CodeBlock code={`mem = Memory(embedding_cache_size=10000)
`} language="python" />

### 4. Working Memory Limits

Keep Working Memory lean. The default maximum is 10,000 entries:

<CodeBlock code={`mem = Memory(
    max_working=10000,  # Maximum working memory entries
    consolidation_interval=1800,  # More frequent consolidation
)
`} language="python" />

### 5. Regular Consolidation

Run consolidation periodically to evict stale entries and maintain query performance:

<CodeBlock code={`# Via Hermes
hermes mnemosyne sleep

# Or programmatically
mem.sleep()
`} language="python" />

## Memory Usage

| Component | Per Entry | 10K Entries |
|-----------|-----------|-------------|
| Working Memory | ~50KB | ~500MB |
| Episodic Memory | ~2KB | ~20MB |
| Semantic Memory | ~500B | ~5MB |
| Vector Index | ~6KB | ~60MB |
| FTS5 Index | ~2KB | ~20MB |

<Callout type="success" title="SSD Recommended">
SQLite performance is heavily I/O bound. Use SSD storage for production deployments. HDD is acceptable for development but will show 5-10x slower query times.
</Callout>
