# Upgrade Guide

Upgrade Mnemosyne and migrate data between versions.

If you're on **v2.7** and want the latest, jump to [Upgrading from v2.7 to v3.0](#upgrading-from-v27-to-v300).

Already on v3.0.x? See [Upgrading to v3.1](#upgrading-to-v31-shared-surface--multilingual-memoria).

## Quick Reference

| What changed | User action |
|---|---|
| New PyPI release | `pip install --upgrade mnemosyne-memory` + restart Hermes |
| Source-only fix | `git pull` + restart Hermes |
| New dependency / entry point | `git pull` + `pip install -e .` + restart Hermes |
| `externally-managed-environment` (Debian/Ubuntu) | Use a venv or `pip install --break-system-packages` |
| SQLite schema changed | See [How to confirm schema changes](#how-to-confirm-schema-changes) |
| E6 TripleStore split (v2.8) | Auto-migrates on first init. Backup at `{db}.pre_e6_backup` |
| MEMORIA architecture (v3.0) | Auto-creates 5 new tables on first init. No manual action needed |
| `plugin.yaml` / tool schema | Restart Hermes only |

## PEP 668: externally-managed-environment on Debian / Ubuntu

Debian 13 (and Ubuntu 24.04+) ship Python with PEP 668 protection. `pip install` outside a virtualenv fails with:

```
error: externally-managed-environment
Ă— This environment is externally managed
```

**Solution 1: Use a virtualenv (recommended)**

<CodeBlock code={`python3 -m venv ~/mnemosyne-venv
source ~/mnemosyne-venv/bin/activate
pip install --upgrade mnemosyne-memory`} language="bash" />

Make sure Hermes is configured to use this venv's Python.

**Solution 2: pipx (for CLI tools)**

```bash
pip install pipx
pipx install mnemosyne-memory
```

**Solution 3: Override (quick fix, use with caution)**

<CodeBlock code={`pip install --upgrade mnemosyne-memory --break-system-packages`} language="bash" />

This bypasses the guard. Fine for personal machines or containers. Not recommended for shared/multi-app systems.

## Upgrading from v2.7 to v3.0

This is the most common jump for existing users. It covers 3 releases worth of changes. Read the relevant sections in order:

1. **v2.7 â†’ v2.8** â€” E6 TripleStore split (schema migration)
2. **v2.8 â†’ v2.9** â€” MCP SDK 1.x compatibility (code only)
3. **v2.9 â†’ v3.0** â€” MEMORIA architecture (new tables)

### Step-by-step

<CodeBlock code={`# 1. Update the package
pip install --upgrade mnemosyne-memory

# (If PEP 668 blocks you, use --break-system-packages)
# pip install --upgrade mnemosyne-memory --break-system-packages

# 2. Restart Hermes to load the new plugin/tools
hermes gateway restart

# 3. Verify
hermes mnemosyne version
# Should show: 3.0.0

hermes mnemosyne stats --global
# Check memory count is preserved

hermes tools list | grep mnemosyne
# Should show 17+ tools`} language="bash" />

**What happens to your data on first run:**

- v2.7 databases get auto-migrated by E6 on first BeamMemory init. Backup written to `{db}.pre_e6_backup`.
- v3.0 creates 5 new MEMORIA tables (`memoria_facts`, `memoria_timelines`, `memoria_instructions`, `memoria_preferences`, `memoria_kg`) via `CREATE TABLE IF NOT EXISTS`. Existing tables are untouched.
- All existing memories, triples, embeddings remain intact.

**If anything goes wrong:**

<CodeBlock code={`# Restore pre-E6 backup
cp ~/.hermes/mnemosyne/data/mnemosyne.db.pre_e6_backup \
   ~/.hermes/mnemosyne/data/mnemosyne.db

# Roll back to v2.7
pip install 'mnemosyne-memory==2.7.0'
hermes gateway restart`} language="bash" />

## Upgrading to v3.1 (Shared Surface & Multilingual MEMORIA)

The v3.1 release adds shared surface memory, multilingual MEMORIA, custom embedding endpoints, and many fixes.

### New capabilities

- **Shared surface memory.** Cross-agent shared persistence via `mnemosyne_shared_*` tools. Each agent gets an isolated shared surface. Activate with `hermes memory surface` commands.
- **Multilingual MEMORIA.** Language auto-detection for German, Russian, and Chinese. Extraction applies language-specific patterns based on detected input language.
- **Custom embedding endpoints.** Configure any OpenAI-compatible embedding provider via `OPENROUTER_BASE_URL` (set to your own server URL). Jina model dimensions auto-detected.
- **Deterministic `get(id)`.** Direct memory retrieval by ID â€” no vector search, no ranking. Call `mnemosyne.get(memory_id)` for exact lookup.

### New environment variables

| Variable | Default | What it does |
|---|---|---|
| `OPENROUTER_BASE_URL` | `https://openrouter.ai/api/v1` | Override the embedding API provider URL |
| `MNEMOSYNE_EMBEDDINGS_VIA_API` | not set | Set to `true` to route all embedding models through the API |

### What to verify after update

<CodeBlock code={`pip install --upgrade mnemosyne-memory
hermes gateway restart

# Verify version
python3 -c "from mnemosyne import __version__; print(__version__)"
# Expected: 3.1.0`} language="bash" />

### Rollback

<CodeBlock code={`pip install 'mnemosyne-memory==3.0.0'
hermes gateway restart`} language="bash" />

Shared surface tables remain in the database but are ignored by v3.0.0.

## Upgrading to v3.1.1 / v3.1.2 (Patch Releases)

These are additive patch releases â€” no schema changes, no manual steps.

### v3.1.1

Renames embedding env vars to the preferred convention (`MNEMOSYNE_EMBEDDING_API_URL` / `MNEMOSYNE_EMBEDDING_API_KEY`). The old names (`OPENROUTER_BASE_URL`, `OPENROUTER_API_KEY`) still work as fallbacks.

<CodeBlock code={`pip install --upgrade mnemosyne-memory
hermes gateway restart`} language="bash" />

### v3.1.2

Strict fact matching is now the default. If you relied on permissive matching (where any query word matched any stored fact), set `MNEMOSYNE_LENIENT_FACT_MATCH=1` to opt back in.

Other fixes: entity prefix similarity now requires 30% min length ratio, single-token strict fact queries (5+ chars, non-stopword) now match instead of being silently rejected.

<CodeBlock code={`# Upgrade
pip install --upgrade mnemosyne-memory
hermes gateway restart

# If recall results seem too narrow after upgrade:
export MNEMOSYNE_LENIENT_FACT_MATCH=1
hermes gateway restart`} language="bash" />

## Schema Migrations

Mnemosyne uses automatic schema migrations that run on first connection after upgrade. There is no manual `migrate` command.

| Version | Migration | Description |
|---|---|---|
| 1.0 â†’ 2.0 | Schema upgrade | Add BEAM tiered memory, FTS5 indices |
| 2.0 â†’ 2.1 | Schema upgrade | Add `session_id` to episodic |
| 2.1 â†’ 2.2 | Schema upgrade | Add temporal graph indexes |
| 2.2 â†’ 2.8 (E6) | `e6_triplestore_split` | Split `triples` into `triples` + `annotations` |
| 2.8 â†’ 3.0 (MEMORIA) | Auto-create | 5 new tables (`memoria_facts`, `memoria_timelines`, `memoria_instructions`, `memoria_preferences`, `memoria_kg`) |

<Callout type="info" title="Automatic Migrations">
Mnemosyne uses automatic schema migrations that run on first connection after an upgrade. There is no `python -m mnemosyne.migrate` command. If migration fails, check logs with `MNEMOSYNE_LOG_LEVEL=debug`.
</Callout>

## Cross-Instance Migration

Export from old instance, import to new using the Python SDK:

<CodeBlock code={`from mnemosyne import Mnemosyne

# Export from old database
old_mem = Mnemosyne(db_path="old.db")
old_mem.export_to_file("export.json")

# Import into new database
new_mem = Mnemosyne(db_path="new.db")
new_mem.import_from_file("export.json")

# Verify
print(new_mem.stats())`} language="python" />

## How to Confirm Schema Changes

Wondering if an update changed the SQLite schema? Here's how to check.

### Before updating (baseline)

<CodeBlock code={`# Dump the current schema
python3 -c "
import sqlite3, pathlib
db = pathlib.Path.home() / '.hermes' / 'mnemosyne' / 'data' / 'mnemosyne.db'
conn = sqlite3.connect(str(db))
schema = conn.execute(\"SELECT sql FROM sqlite_master WHERE type='table' ORDER BY name\").fetchall()
for row in schema:
    print(row[0] + ';')
conn.close()
" > ~/mnemosyne_schema_baseline.txt`} language="bash" />

### After updating (compare)

<CodeBlock code={`# Dump the new schema
python3 -c "
import sqlite3, pathlib
db = pathlib.Path.home() / '.hermes' / 'mnemosyne' / 'data' / 'mnemosyne.db'
conn = sqlite3.connect(str(db))
schema = conn.execute(\"SELECT sql FROM sqlite_master WHERE type='table' ORDER BY name\").fetchall()
for row in schema:
    print(row[0] + ';')
conn.close()
" > ~/mnemosyne_schema_new.txt

# Compare
diff ~/mnemosyne_schema_baseline.txt ~/mnemosyne_schema_new.txt`} language="bash" />

### Quick check: what version was my DB created by?

<CodeBlock code={`python3 -c "
import sqlite3, pathlib
db = pathlib.Path.home() / '.hermes' / 'mnemosyne' / 'data' / 'mnemosyne.db'
conn = sqlite3.connect(str(db))
tables = conn.execute(\"SELECT name FROM sqlite_master WHERE type='table' ORDER BY name\").fetchall()
names = [t[0] for t in tables]
if 'memoria_facts' in names:
    print('DB schema: v3.0+ (MEMORIA)')
elif 'annotations' in names:
    print('DB schema: v2.8+ (E6 TripleStore split)')
elif 'episodic_memory' in names:
    print('DB schema: v2.0+ (BEAM)')
else:
    print('DB schema: v1.x (legacy)')
conn.close()
"`} language="bash" />

## Database Migrations

Mnemosyne uses `CREATE TABLE IF NOT EXISTS` and `CREATE INDEX IF NOT EXISTS`, so most schema changes upgrade automatically.

Run a migration script only when:
- The CHANGELOG explicitly mentions a database schema change
- You are upgrading from a pre-2.0 version
- You see errors about missing columns or tables

### Available migration scripts

| Script | What it does |
|---|---|
| `scripts/migrate_from_legacy.py` | Migrates from v1.x ephemeral databases to the canonical v2+ path. Idempotent. |
| `scripts/migrate_triplestore_split.py` | Manual E6 migration (v2.8). Only needed if you set `MNEMOSYNE_AUTO_MIGRATE=0`. Idempotent. |

### E6 TripleStore Split

<CodeBlock code={`pip install --upgrade mnemosyne-memory
hermes gateway restart
# Check logs for:
#   "E6: auto-migrated N annotation rows from triples -> annotations."`} language="bash" />

**Manual migration (explicit control):**

<CodeBlock code={`export MNEMOSYNE_AUTO_MIGRATE=0
hermes gateway restart

# Preview
python scripts/migrate_triplestore_split.py --dry-run

# Apply
python scripts/migrate_triplestore_split.py`} language="bash" />

**What to verify after E6 migration:**

<CodeBlock code={`# Check annotations table exists
python3 -c "
import sqlite3, pathlib
db = pathlib.Path.home() / '.hermes' / 'mnemosyne' / 'data' / 'mnemosyne.db'
conn = sqlite3.connect(str(db))
count = conn.execute('SELECT COUNT(*) FROM annotations').fetchone()[0]
print(f'Annotations table has {count} rows')
conn.close()
"`} language="bash" />

All migration scripts are idempotent â€” safe to run multiple times.

## Rollback

### Roll back to a specific version

<CodeBlock code={`# Pin to a known good version
pip install 'mnemosyne-memory==2.7.0'

# Or from source
cd mnemosyne
git checkout v2.7.0
pip install -e .

# Restart Hermes
hermes gateway restart`} language="bash" />

### Restore a database backup

If you have a DB backup from before the update:

<CodeBlock code={`# E6 auto-backup
cp ~/.hermes/mnemosyne/data/mnemosyne.db.pre_e6_backup \
   ~/.hermes/mnemosyne/data/mnemosyne.db

# Or any custom backup
cp ~/backups/mnemosyne_20260101.db \
   ~/.hermes/mnemosyne/data/mnemosyne.db`} language="bash" />

### Export, nuke, re-import

<CodeBlock code={`# Export current data
hermes mnemosyne export --output ~/backup.json

# Delete the database entirely
rm ~/.hermes/mnemosyne/data/mnemosyne.db

# Start fresh with old version
pip install 'mnemosyne-memory==2.7.0'
hermes gateway restart

# Re-import
hermes mnemosyne import --input ~/backup.json`} language="bash" />

## Troubleshooting

### "Command not found" after update

Entry points are registered at install time, not at runtime. Re-run the install:

<CodeBlock code={`pip install -e .`} language="bash" />

### "No module named mnemosyne" after update

Your virtual environment may have been deactivated or the editable install broke. Re-install:

<CodeBlock code={`pip install -e .`} language="bash" />

### Plugin changes not taking effect

Hermes caches plugins at startup. You must restart:

<CodeBlock code={`hermes gateway restart`} language="bash" />

### "externally-managed-environment" errors

You're on Debian 13+ / Ubuntu 24.04+ (PEP 668). See the [PEP 668 section](#pep-668-externally-managed-environment-on-debian--ubuntu).

### Database errors after schema change

If you see errors about missing columns or tables:

<CodeBlock code={`# Try auto-repair by restarting
hermes gateway restart

# If that fails, run the legacy migration
python scripts/migrate_from_legacy.py

# If errors persist, export, delete, re-import
hermes mnemosyne export --output ~/backup.json
rm ~/.hermes/mnemosyne/data/mnemosyne.db
hermes mnemosyne import --input ~/backup.json`} language="bash" />

### Memory count dropped after update

The E6 migration moves annotation rows from `triples` to a new `annotations` table. This does not delete memories.

<CodeBlock code={`hermes mnemosyne stats --global

# Check E6 migration log
grep -i "auto-migrated\|E6" ~/.hermes/logs/gateway.log`} language="bash" />
