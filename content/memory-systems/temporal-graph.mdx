# Temporal Knowledge Graph

The graph structure underlying Semantic Memory, enabling relationship traversal and time-aware queries.

## Graph Structure

<MermaidDiagram>

graph LR
    A[Alice] -->|prefers_language| B[Go]
    A -->|works_on| C[Project Alpha]
    C -->|has_deadline| D[2026-05-01]
    C -->|depends_on| E[Auth Service]
    E -->|maintained_by| F[Bob]

</MermaidDiagram>

## Temporal Dimensions

Every edge has temporal metadata:

- **`valid_from`**: When the relationship became true
- **`valid_until`**: When it ceased being true (NULL = current)
- **`confidence`**: How certain we are about this fact
- **`source`**: Which memory entry this was extracted from

## Query Patterns

### Direct Neighbors

<CodeBlock code={`# Who works on Project Alpha?
mem.query_facts(predicate="works_on", object="Project Alpha")
# &gt; Alice
`} language="python" />

### Path Traversal

<CodeBlock code={`# What services does Project Alpha depend on (2 hops)?
mem.traverse(subject="Project Alpha", depth=2)
# &gt; Auth Service &gt; Bob (maintainer)
`} language="python" />

### Temporal Filtering

<CodeBlock code={`# What were Alice's preferences as of March 1?
mem.query_facts(
    subject="Alice",
    predicate="prefers_language",
    at_time="2026-03-01",
)
`} language="python" />

## Implementation

The graph is stored in SQLite with:

- **Nodes table**: Entity lookup and deduplication
- **Edges table**: Triples with temporal validity
- **Indices**: Subject, predicate, object, and time-range indexes

No external graph database required.

<Callout type="success" title="Performance Tip">
Graph queries are fast for shallow traversals (1–3 hops). Deep traversals (5+ hops) may require optimization. Consider materializing common paths if your use case requires deep graph walks.
</Callout>
