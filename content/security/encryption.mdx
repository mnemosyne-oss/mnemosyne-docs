# Encryption

Mnemosyne provides **optional, first-class client-side encryption** for sync payloads. When enabled, memory content is encrypted before it leaves your machine and can only be decrypted by instances holding the same key.

## Data at Rest

All data is stored in a local SQLite file (`~/.hermes/mnemosyne/data/mnemosyne.db`). Mnemosyne does not encrypt the database file at rest — this is delegated to your OS:

| Layer | Data | Security |
|-------|------|----------|
| **Working memory** | Recent agent context, preferences, facts | Plaintext in SQLite. File permissions control OS-level access. |
| **Episodic memory** | Consolidated long-term memories, vector embeddings | Same. Embeddings are mathematical vectors, not reversible to original text. |
| **TripleStore** | Temporal knowledge graph (subject-predicate-object triples) | Same. |
| **Config** | API keys, endpoints, encryption settings | Stored in environment variables or config.yaml. Never logged or transmitted. |

**Recommendations:**

- `chmod 600 ~/.hermes/mnemosyne/data/mnemosyne.db`
- Full-disk encryption (LUKS, FileVault, BitLocker)
- Memory bank isolation for shared machines: `mnemosyne --bank <name>`

## Data in Transit (Sync)

When `mnemosyne sync` is enabled, memory content travels between instances over HTTP/HTTPS.

### Without Client-Side Encryption

| What the remote server sees | Example |
|-----------------------------|---------|
| Memory content (plaintext) | `"User prefers dark mode"` |
| Importance scores | `0.9` |
| Memory type / source | `"preference"`, `"cli"` |
| Event metadata | `event_id`, `timestamp`, `device_id`, `operation` |
| Vector embeddings | Opaque binary (not reversible to text, but similarity-analysis possible) |

**Protection:** TLS encryption between instances. Recommended: reverse proxy (Caddy, Nginx, Traefik) with automatic HTTPS.

### With Client-Side Encryption (`--encrypt`)

| What the remote server sees | Example |
|-----------------------------|---------|
| Memory content | `Encrypted::<base64 ciphertext>` (unreadable without the key) |
| Importance scores | Encrypted alongside content |
| Memory type / source | Encrypted alongside content |
| **Unencrypted metadata** (required for routing) | `event_id`, `timestamp`, `device_id`, `operation` |
| Vector embeddings | Encrypted as part of payload |

**Protection:** Authenticated encryption via `cryptography.fernet.Fernet` (AES-CBC) or `nacl.secret.SecretBox` (XSalsa20-Poly1305). The remote server cannot decrypt any payload content.

## Client-Side Encryption in Detail

### How It Works

```
┌─────────────────────┐           ┌─────────────────────┐
│   Local Instance    │           │   Remote Instance   │
│                     │           │                     │
│  1. Encrypt payload │  ──────►  │  3. Store encrypted │
│     with local key  │  (TLS)   │     payload as-is   │
│                     │           │                     │
│  4. Request changes │  ◄────── │  2. Return encrypted │
│  5. Decrypt payload │  (TLS)   │     payloads         │
│     with local key  │           │                     │
└─────────────────────┘           └─────────────────────┘
```

The encryption key never leaves the local instance. The remote instance stores and serves encrypted payloads without ever decrypting them.

### Encryption Backend

`SyncEncryption` tries these backends in order:

1. **`cryptography.fernet.Fernet`** — AES-128-CBC with HMAC-SHA256 (requires `cryptography>=41.0`)
2. **`nacl.secret.SecretBox`** — XSalsa20-Poly1305 (requires `PyNaCl>=1.5`)

Install either with: `pip install mnemosyne-memory[sync]`

### Key Sources

| Source | Configuration | Where Supported |
|--------|---------------|-----------------|
| **Environment variable** | `MNEMOSYNE_SYNC_KEY=<base64-key>` | Core `SyncEncryption`, Hermes adapter |
| **File path** | `--encrypt /path/to/keyfile` | CLI, Hermes adapter |
| **Raw key string** | `--encrypt <base64-key>` | CLI, Hermes adapter |
| **OS keyring** | `MNEMOSYNE_SYNC_KEY_SOURCE=keyring` | Hermes sync adapter only |
| **Prompt** | `MNEMOSYNE_SYNC_KEY_SOURCE=prompt` | Hermes sync adapter only |

Core `SyncEncryption.from_config()` reads from `$MNEMOSYNE_SYNC_KEY` or a provided key source. The Hermes sync adapter (`hermes_memory_provider/sync_adapter.py`) supports additional sources through `MNEMOSYNE_SYNC_KEY_SOURCE`.

### Key Derivation

`SyncEncryption.derive_key()` converts a passphrase to a 32-byte key:

- **Default:** PBKDF2-HMAC-SHA256 with 600,000 iterations
- **If argon2-cffi is installed:** Argon2id (memory-hard, recommended)

Note: `from_config()` does not currently call `derive_key()` — passphrases are not yet supported as a key source in core. Use a pre-derived key or the Hermes adapter for passphrase support.

```bash
# Generate a random key
mnemosyne sync generate-key
# Output: MNEMOSYNE_SYNC_KEY=7A8B3C... (base64, 32 bytes)

# Use with CLI
mnemosyne sync --encrypt MNEMOSYNE_SYNC_KEY=7A8B3C...
mnemosyne sync --encrypt /path/to/keyfile

# Enable in Hermes (config.yaml)
mnemosyne:
  sync:
    key: "7A8B3C..."
    encrypt: true
```

### What Gets Encrypted

The **payload** field of each sync event:

- Memory `content` text
- `importance` score
- `source` field
- `metadata_json` (if present)
- `memory_type` / `veracity` fields
- Vector embeddings (`binary_vector`)

### What Stays Unencrypted (Metadata)

These fields are always plaintext (needed for routing and conflict resolution):

- `event_id` — UUID
- `memory_id` — References the memory row
- `operation` — CREATE / UPDATE / DELETE / CONSOLIDATE
- `timestamp` — ISO 8601
- `device_id` — Which device originated the event
- `parent_event_ids` — Causality chain

An adversary with access to the remote sync server can learn *how many* events and *when*, but not their content.

## CLI Commands

| Command | Description |
|---------|-------------|
| `mnemosyne sync generate-key` | Generate a random 32-byte base64-encoded key |
| `mnemosyne sync --encrypt <key\|file>` | Sync with encryption enabled |

## Threat Model

### In Scope

| Threat | Mitigation |
|--------|-----------|
| Remote server compromise -> memory content exposure | Client-side encryption renders payloads unreadable |
| TLS interception (MITM) | TLS with certificate validation |
| Unauthorized sync access | API key / JWT authentication on sync endpoints |
| Replay attacks | Unique event IDs + timestamp validation |
| Sync of malicious content | Destination-side content filtering via `ignore_patterns` |

### Out of Scope

| Threat | Rationale |
|--------|----------|
| Side-channel attacks on encryption timing | Performance-sensitive sync makes constant-time hard. Use TLS for transit. |
| Quantum cryptanalysis of Fernet/SecretBox | Not a practical threat for memory content. Key rotation mitigates long-term risk. |
| Physical access to the database file | OS-level controls (file permissions, disk encryption) are the user's responsibility |
| Denial of service on sync endpoint | Self-hosted deployments manage their own rate limiting |

## Best Practices

1. **Always use HTTPS/TLS** for sync connections.
2. **Enable client-side encryption** when syncing over the internet or to a shared VPS.
3. **Rotate sync keys periodically** (every 90 days).
4. **Back up encryption keys** separately from your Mnemosyne database. Losing the key means losing access.
5. **Use environment-specific keys** — separate keys for dev, staging, production.
6. **Monitor sync events** with `mnemosyne sync status --remote <url>`.
